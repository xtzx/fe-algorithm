### 参与方式

```

├── 修复文档错误（typo/翻译）
├── 改进 README
├── 回答 Issue 问题
└── 提交 Bug 报告

进阶级（3-6 个月）：
├── 修复简单 Bug

高级（6 个月+）：
├── 实现新功能
├── 性能优化
├── 架构改进
└── 担任 Maintainer
```

### 提交第一个 PR 的流程

```bash
# 1. Fork 项目到自己账号
# 在 GitHub 网页上点击 Fork

# 2. 克隆到本地
git clone https://github.com/YOUR_USERNAME/project.git
cd project

# 3. 添加上游仓库
git remote add upstream https://github.com/ORIGINAL/project.git

# 4. 创建分支
git checkout -b fix-typo-readme

# 5. 做出修改
# ... 编辑文件 ...

# 6. 提交
git add .
git commit -m "docs: fix typo in README"

# 7. 推送
git push origin fix-typo-readme

# 8. 在 GitHub 上创建 Pull Request
```

### PR 标题规范

```
类型：描述（Issue 编号）

类型包括：
- feat: 新功能
- fix: Bug 修复
- docs: 文档
- style: 格式
- refactor: 重构
- test: 测试
- chore: 构建/工具

示例：
- docs: fix typo in README
- fix: handle empty input in RAG pipeline (#123)
- feat: add support for streaming response
```

## 技术博客运营

### 平台选择

```
国内平台：
- 掘金：技术社区活跃，曝光高
- 知乎：专栏影响力大
- CSDN：SEO 好，搜索流量大
- 公众号：私域流量，传播性好

国外平台：
- Medium：国际影响力
- Dev.to：开发者社区
- Hashnode：个人博客托管

自建博客：
- GitHub Pages + Hugo/Hexo
- Notion 公开页面
- 自建站点
```

### 内容类型

```
1. 项目复盘（最有价值）
   - 项目背景和目标
   - 技术选型和原因
   - 核心实现细节
   - 遇到的问题和解决
   - 量化成果和反思

2. 技术教程
   - 入门指南
   - 最佳实践
   - 对比评测

3. 踩坑总结
   - 问题描述
   - 排查过程
   - 解决方案
   - 经验教训

4. 源码分析
   - 框架原理
   - 核心实现
   - 设计思想

5. 行业观察
   - 技术趋势
   - 工具对比
   - 未来展望
```

### 写作节奏建议

```
初期（1-3 个月）：
- 每月 1-2 篇
- 主要写学习笔记和踩坑记录
- 不追求完美，重在坚持

中期（3-6 个月）：
- 每月 2-3 篇
- 增加项目复盘和教程
- 注重内容质量

长期：
- 每月 2-4 篇
- 形成系列内容
- 建立个人风格
```
